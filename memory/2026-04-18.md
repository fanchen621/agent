# 2026-04-18 记忆沉淀

## Twitter 爬取方案全量测试结论（2026-04-17 夜）

### 已验证可用的完整链路
```
nitter RSS → Jina.ai → pbs.twimg.com直连
```
- nitter.net/{user}/rss：获取时间线+推文ID（每次18条）
- r.jina.ai/https://x.com/{user}/status/{id}：提取完整文本+图片URL
- pbs.twimg.com/media/...：直接下载图片，腾讯云/VPS均通
- 脚本：/root/fetch_tweets.py（VPS上已验证运行成功）

### 测试失败的方法（已放弃）
| 方法 | 状态 | 原因 |
|------|------|------|
| syndication via WARP | ❌ 不稳定 | JSON/HTML动态切换 |
| GraphQL Query ID | ❌ 全部404 | Twitter轮换端点，测了10个ID |
| foxpy.cc.cd | ⚠️ 部分可用 | Bearer Token失效，只返回URL片段 |
| vxtwitter.com | ❌ 404 | 已下线 |

| SOCKS5 7891→Twitter | ❌ HTTP 000 | 数据中心IP封禁 |

### 关键发现
1. **Bearer Token 自动刷新可行**：TwitterAndroid/8.0.0 UA + Bearer Token → guest_token 成功获取，无需手动
2. **syndication不稳定**：Twitter动态切换JSON/HTML，不可依赖
3. **GraphQL Query ID 每日轮换**：硬编码方案不可行
4. **媒体视频提取**：需要JS渲染，服务器端无法完成

### DuckDNS DNS覆盖问题
- Twitter Proxy Worker 部署在 twitter-proxy.flw10088.duckdns.org
- DuckDNS 动态DNS会覆盖Worker路由的DNS记录（将A record从192.0.2.1改回Oracle Cloud IP）
- 导致Worker无法通过自定义域名访问
- 解决方案：用户需修复DNS，或使用CF默认域名
