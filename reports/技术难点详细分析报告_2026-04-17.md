## P2-1｜共识验证层（source_consensus）

### 前因
DragonFlow的**健康检查系统**有6个检查项，其中`source_consensus`负责验证「不同数据源之间的数据一致性」。

### 问题现象
```
[source_consensus] status: warn
  index: 尚无质量样本（source=tencent, count=0, validated_count=0）
  stock: 尚无质量样本（source=sina, count=0, validated_count=0）
```

### 根因分析
`soruce_consensus`的核心逻辑位于`selfheal/diagnostics.py`的`check_source_consensus()`函数。它需要：
1. 从多个数据源（tencent/sina/eastmoney）获取**同一批股票**的同一批字段
2. 对比各源数据，验证gap是否在容忍范围内
3. 计算平均置信度

**根因**：系统从未有过**质量样本积累**。追踪代码发现`check_source_consensus`调用了`_validate_stocks()`函数，该函数依赖`resilient.py`中维护的`_last_good_stocks`缓存。但从腾讯云部署以来：
- 雪球爬虫长期样本为0（Cookie失效）
- Sina和Tencent的数据从未被consensus层正式记录过

**次级根因**：共识验证需要主动写入`validated_count`，但`MarketSnapshot`表中从未创建过`sentiment`类型的快照（确认：`sentiment: 0 records`）。

### 后果
- 健康检查永远显示warn，无法真实反映数据质量
- 当数据源失效时，系统无法自动判断哪个源可信

### 解决路径
在`sentiment`模块或`market_snapshot`模块中，**每次成功获取多源数据时，记录共识快照**，触发consensus计算。实现量级：约50行代码。

## P2-2｜maybe_refresh耗时85秒（realtime_watch慢速）

### 前因
`autopilot`每55秒调用一次`maybe_refresh()`，期望在60秒内完成。但实测需要**85秒**，导致SIGKILL。

### 问题现象
```
[realtime_watch] 0 live / 0 evolution candidates from 27 symbols in 85288.7ms
```
仅处理27个股票（board_map里的涨停股），耗时85秒。

### 根因分析——完整调用链
```
autopilot._refresh_live_watch()
  → maybe_refresh()
    → _load_board_map()        ✅ 0.4秒（25个涨停股）
    → _load_db_context()        ✅ 0.0秒（无持仓/观察列表）
    → _load_market_rows()        ✅ 3.2秒（Sina VIP 10页）
    → _load_feature_map()         ❌ 70-80秒
        → _get_cached_feature() × 27股票
            → compute_stock_features()
                → get_stock_history()
                    → Tencent K线: 0.1秒 ✅
                    → **BaoStock换手率补充**: 2-3秒/股 ❌
```

**关键发现**：`_supplement_turnover_from_baostock()` 对每只股票调用**BaoStock查询**，BaoStock需要登录+查询，每次2-3秒。27个股票 × 2秒 = 54秒，叠加网络延迟≈70-80秒。

### 为什么是27个股票
因为`board_map`有25个涨停股，`watch_entries`和`positions`都是空的，所以`focus_codes`只包含25个board股。`spot_limit`默认是`80`，但`priority_codes`只有25个，最终处理27个（board + Sina前2页最高涨幅股）。

### 后果
- autopilot在55秒间隔内无法完成刷新
- quick_scan候选被覆盖（已通过circuit breaker缓解）
- SIGKILL导致数据流中断

### 解决路径
1. **紧急**：在`_supplement_turnover_from_baostock`加缓存（已有60-120秒TTL），避免重复查询
2. **中期**：将`compute_stock_features`中的BaoStock调用改为异步并行（ThreadPoolExecutor）
3. **根本**：realtime_watch应仅处理`watchlist`中的持仓股，不应处理全市场

## P2-3｜XHS Cookie过期（web_session失效）

### 前因
小红书爬虫依赖Cookie中的`web_session`字段进行身份认证。

### 问题现象
```
[舆情] source crawl failed: name '_HOT_STOCK_BARS' is not defined
```
雪球Playwright正常工作（7条样本），但舆情源只有雪球。

### 根因分析
XHS Cookie的`a1`值已获取，但`web_session`过期（471秒前）。XHS的数据流：
```
xhs_data.py → _hot_noteBars() → web_session无效 → 401/403
```

### 解决路径
在Playwright中重新登录XHS，提取新的`web_session`值。或者在浏览器端手动获取新Cookie。

## P3-4｜Twitter视频缺失OAuth Write权限

### 前因
Twitter API对视频上传有严格的OAuth Write权限要求。当前系统只有游客Token（读权限），无法发视频。

### 问题现象
推文发布功能正常（文字+图片），但视频上传返回403。

### 根因
Twitter API v2对`media/upload`端点需要：
- Write权限的OAuth 1.0a或OAuth 2.0 Bearer Token
- 当前只有`auth_token`和`ct0` Cookie（游客模式）

### 解决路径
1. 申请Twitter Developer账号并创建App
2. 获取OAuth 2.0 Bearer Token（无用户上下文）
3. 对视频上传使用`media/upload` API

## P3-5｜MiniMax-VL端点404（API Key失效）

### 前因
MiniMax旧API端点`api.minimax.chat/v1/vision`已失效。

### 问题现象
```
GET https://api.minimax.chat/v1/vision → 404 page not found
```

### 根因
从VPS测试发现：
```
GET https://www.minimaxi.com/v1 → 404 page not found
GET https://api.minimaxi.com → 308 redirect
```
说明MiniMax API已迁移到`api.minimaxi.com`域名，旧端点已下线。两台机器都没有有效API Key。

### 解决路径
1. 在MiniMax开放平台（platform.minimaxi.com）重新注册/获取有效Key
2. 更新OpenClaw配置中的`agents.defaults.imageModel`为`minimaxi/xxx`
3. 确认新端点路径（可能从`/v1/vision`变为其他路径）
